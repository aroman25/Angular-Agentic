The Modern Angular v21 Development Paradigm: A Technical Manual for High-Performance, Accessible Applications
Orchestrator Compatibility Note: This blueprint is v21-first, but orchestrated app generation and validation must remain compatible with Angular v20 and v21 unless the user story or template explicitly narrows the target.
1. The Strategic Shift: Angular v21 and the Signals-First Era
The evolution of Angular from version 16 through version 21 represents the most significant architectural pivot in the framework's history. While version 20 marked the stabilization of the Signals primitive, version 21 defines the "Zoneless by Default" milestone. This shift moves the framework away from the legacy of implicit reactivity—where the framework guessed when to update the UI via global monkey-patching—toward an explicit, deterministic model. Version 21 represents a stabilized era where the architectural choice of Zoneless execution is no longer an experiment but the recommended path for production-grade applications.This transition necessitates a move from the "Legacy Monolith" (Zone.js) to the "Modern Node" (Zoneless/Signals) architecture.| Feature | Legacy Monolith (Zone.js) | Modern Node (Zoneless/Signals) | Technical Impact || ------ | ------ | ------ | ------ || Change Detection | Implicit (Global "Monkey-patching") | Explicit (Fine-grained Signals) | 30-40% improvement in runtime performance || Bundle Size | Includes ~30KB Zone.js overhead | Eliminates Zone.js dependency | Faster initial load; -30KB compressed weight || Core Web Vitals | Baseline LCP/INP | 40–50% LCP Improvement | Surgical updates reduce main-thread blocking || Memory Usage | Higher (Global tracking) | 15–25% Reduction | Efficient execution in data-intensive apps |
This shift necessitates a complete rethink of application state. By removing the "black box" of automatic change detection, architects now gain surgical control over the rendering pipeline, transitioning into a technical setup designed for sub-60fps UI performance.

3. Core Architecture: Mastering Signals and Zoneless Reactivity
The "Mechanical Shift" from implicit monkey-patching to explicit reactivity is the strategic secret to achieving high-performance UIs. In the legacy paradigm, Zone.js intercepted every asynchronous event to trigger global change detection. In v21, reactivity is deterministic: Angular only updates the specific DOM nodes that depend on a modified Signal.Functional Roles of Stable Signal APIs:
signal : The core reactive primitive.
So What?  Provides synchronous state management without the cognitive overhead of RxJS Observables.
computed : Derived reactive state.
So What?  Memorizes values and only re-evaluates when dependencies change, preventing expensive re-computations in the template.
effect : Side-effect execution.
So What?  Stabilized in v21 with predictable cleanup and execution timing, replacing legacy lifecycle hooks for external integrations.
linkedSignal : A writable signal deriving its state from another source.
So What?  Eliminates manual reset logic in derived state; it automatically resets its internal value when its source signal updates.The "Zoneless Revolution" complements this by removing 30KB of bundle weight and ending the era of "black-box" change detection. Furthermore, v21 marks the stabilization of afterEveryRender and afterNextRender, providing a reliable replacement for the legacy afterRender logic. This deterministic approach manifests most clearly in the implementation of accessible, high-performance UI components.
4. The UI Layer: Headless Accessibility with Angular Aria and Tailwind CSS
Architects must embrace "Headless UI" to decouple accessibility logic (Angular Aria) from visual styling (Tailwind). This provides design freedom without sacrificing WCAG compliance. Angular Aria provides unstyled primitives (Tabs, Accordions, Grids) that manage complex ARIA roles and keyboard interactions, while Tailwind mitigates "CSS Leaks" in micro-frontend contexts through scoped utilities.Instructional Guide: Accessible Tab System (Mechanical Shift):  Standard ARIA patterns often force keyboard users through every tab. As a Senior Architect, we override this by making  all tabs TABable , allowing users to skip a tablist without being forced through arrow-key navigation.
Agent Reuse Rule: Before building new UI, agents must inspect `src/app/shared/ui/index.ts` and reuse existing `shared/ui` components first. Each component file includes a top usage comment documenting selector, inputs/models/outputs, and content slots; agents should read that comment before implementation.
Template-First Optimization Rule: When a defect is reusable across generated apps (template config, shared UI import paths, shared component bug, boilerplate app shell issue), fix the template and orchestration guidance/checks upstream before repeating one-off fixes in generated apps.
Work Order Data Fidelity Rule: Planning agents must include a "User Story Data Points" section and a requirement traceability mapping so routes, field names/options, validation constraints, and required selectors are copied from the user story (assumptions must be labeled).
Shared UI Runtime DI Rule: For Angular Aria-based shared wrappers that project child components with `<ng-content>`, attach provider/group directives to the shared component host (or otherwise guarantee provider visibility) to avoid runtime NG0201 injection errors.
Forms Agent Rule: For form-centric features, agents must use typed Reactive Forms with explicit validators, render validation feedback in the template, and bridge non-CVA `shared/ui` controls through `value`/`valueChange` rather than `formControlName`.
Agent Pattern Catalog Rule: Canonical form/layout examples live in template docs under `automate-angular-template/docs/agent-patterns/` (`README.md`, `forms-core.md`, `layouts-core.md`). Planning agents should cite selected pattern IDs in the Work Order `UI/UX Requirements` using a `Template Pattern References:` line, and any deviation from the catalog should be justified with `Deviation Notes`.
Orchestrator Skill Pack Rule: Planner/Developer/Validator prompts should incorporate the local orchestrator skill pack (`orchestrator/skills/angular-orchestrator-v20-v21/`) derived from Codex Angular skills to keep form, accessibility, Tailwind, routing, and validation guidance consistent across complex stories.

5. Advanced Data Handling: Signal-Based Forms vs. Legacy Reactive Forms
Angular v21 introduces  Signal Forms  (currently in Developer Preview), which utilize a  FieldTree Architecture . This model is the superior successor to Reactive Forms because it aligns form state with the framework’s core reactivity model, removing the verbose requirement for ControlValueAccessor (CVA).Architectural Comparison: Legacy vs. Signal Forms| Feature | Legacy Reactive Forms | Experimental Signal Forms (v21+) || ------ | ------ | ------ || State Management | RxJS Observables (valueChanges) | Native Signals || Type Safety | Often requires manual definitions | Deep, automatic type inference || Boilerplate | High (CVA requires 30+ lines) | Low (Direct binding in ~3 lines) || Performance | Tree-based scanning | Fine-grained, surgical updates |
Schema-Based Validation:  Implementation in v21 uses the form() and validate functions. Custom controls no longer require the heavy boilerplate of CVA; they bind directly to the signal-based FieldTree. Synchronous and asynchronous logic are handled within a unified validate function, providing a cleaner integration for server-side validation than complex RxJS streams.

7. Documented Intention: Folder Structure and Project Architecture
The v21 paradigm favors "Intention-Oriented" structures where suffixes like .component.ts are optional and folder location defines purpose. This structure is critical for vertical slicing and mitigating CSS leaks.

├── features/               # Domain-driven Vertical Slices
